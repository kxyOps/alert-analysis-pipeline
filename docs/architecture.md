## 数据流

### 在线告警链路（Grafana → 飞书）

```
Grafana 触发告警
    │ POST webhook
    ▼
n8n Webhook 节点
    │ Code 节点: 采集相关容器日志（Docker API）
    │ Set 节点: JSON.stringify payload
    │ Crypto 节点: HMAC-SHA256 签名
    ▼
Hermes Gateway (HTTP POST + X-Webhook-Signature)
    │ 验证签名，分发到 webhook
    ▼
Hermes Agent
    │ 1) kb.py match --json '<payload>' → 查知识库
    │ 2) 命中 → 直接输出已知故障的根因和恢复步骤
    │ 3) 未命中 → alert-analyzer 分析，给出排查建议
    ▼
飞书群通知
```

### 知识库更新链路（飞书 → KB）

```
用户在飞书群 @bot
    │ 发送: fault <告警消息> <根因ID> [备注]
    ▼
Hermes Agent (通过 channel_prompt)
    │ terminal 执行: kb.py add "<消息>" "<根因ID>" "[备注]"
    ▼
KB JSON 写入 → KB Server SQLite 同步
    │ Web UI 实时可见
    ▼
cron 每周清理过期记录（保留最近 150 条）
```

## 组件说明

| 组件 | 端口 | 功能 |
|------|------|------|
| Grafana | 3000 | 仪表盘 + 告警规则配置 |
| Prometheus | 9090 | 指标采集和告警评估 |
| n8n | 5678 | 工作流引擎，编排告警转发 |
| KB Server | 8888 | 知识库 Web UI + REST API |
| Hermes | 9091 | Agent Gateway，接收 webhook |
| 飞书 | - | 接收告警通知、@bot 添加记录 |

## 核心设计要点

1. **HMAC 签名验证** — n8n → Hermes 之间使用 HMAC-SHA256 签名，防止伪造请求
2. **知识库多字段匹配** — alertname + message 双字段正则匹配，支持排除规则（如 resolved/recover）
3. **零外部依赖** — kb.py 和 kb-server.py 仅使用 Python 标准库
4. **飞书 @bot 输入** — 通过 channel_prompt 让 agent 执行 kb.py add，用户无需登录服务器
5. **定期清理** — cron 每周清理旧记录，防止 KB 无限膨胀
